# Human Vs AI Code Detection, Task A: Methodology Report

## Final Score: **0.89382** (Public & Private)

---

## 1. Task Overview

Task A is a binary classification problem: given a code snippet, determine whether it was written by a **human (label 0)** or generated by an **AI model (label 1)**. The competition metric is **macro F1**, which equally weights performance on both classes regardless of class imbalance.

---

## 2. Dataset

### 2.1 Source

Final training and evaluation used a curated dataset provided as part of the SemEval 2026 Task 13 competition. This dataset differs significantly from the raw competition files used in earlier attempts in terms of language diversity, generator coverage, and class balance.

### 2.2 Structure

Each parquet file contains four columns:

| Column | Type | Description |
|---|---|---|
| `code` | string | Raw source code snippet |
| `label` | int | 0 = Human, 1 = AI-generated |
| `language` | string | Programming language |
| `generator` | string | Source model or "Human" |

### 2.3 Splits

| Split | Samples | Human | AI | Languages |
|---|---|---|---|---|
| Train | 50,332 | 25,166 (50%) | 25,166 (50%) | 8 |
| Validation | 9,882 | 4,941 (50%) | 4,941 (50%) | 8 |
| Test Sample | 1,000 | 777 (77.7%) | 223 (22.3%) | 8 |

### 2.4 Language Distribution (Training)

| Language | Samples | % of Total |
|---|---|---|
| Python | 7,804 | 15.5% |
| Java | 7,804 | 15.5% |
| C# | 7,784 | 15.5% |
| C++ | 7,622 | 15.1% |
| JavaScript | 7,510 | 14.9% |
| Go | 5,624 | 11.2% |
| C | 4,914 | 9.8% |
| PHP | 1,270 | 2.5% |

Languages are **roughly equally represented**, with PHP being the only notable minority. This near-uniform distribution is by design and eliminates the need for any language-based resampling.

### 2.5 AI Generator Diversity (Training)

The AI-generated samples (25,166 total) come from **30+ distinct models**, including:

- **GPT-4o** (6,742 samples — largest single source)
- Meta Llama 3.1/3.2 (8B, 11B variants)
- Mistral 7B, Devstral Small
- IBM Granite 3.2 / 3.3
- DeepSeek V3, DeepSeek R1
- Qwen 2.5 Coder (1.5B, 14B, 72B)
- Microsoft Phi-3 (mini, small, medium)
- Google Gemma 3
- BigCode StarCoder, StarCoder2-3B, StarCoder2-15B
- And others

This diversity is critical: the model cannot overfit to the stylistic patterns of any single generator.

### 2.6 Generator Shift at Test Time

The test sample introduces **entirely unseen generators** not present during training, including:

- Llama 4 Scout / Maverick
- Gemini 2.0 Flash, 2.5 Flash
- Phi-4, Phi-4 Multimodal
- Mistral Small 24B, Mistral Nemo
- CodeLlama, DeepSeek Coder variants

This is an intentional generalization challenge. A model that memorizes generator-specific signatures will fail; a model that learns the underlying structural differences between human and AI code will succeed.

## 3. Model Architecture

A hybrid classifier combining a **frozen LLM backbone** with **handcrafted stylometric features**.

```
backbone_embedding [2048]  ──► LayerNorm ──► tanh-Linear ──► [2048]
handcrafted_features [102] ──► MLP (102 → 256 → 128) ──► [128]
                                                               │
                                                  cat [2176] ──► Fusion MLP ──► [2]
```

### 3.1 Backbone: StarCoder2-3B (Frozen)

- **Model**: `bigcode/starcoder2-3b`, loaded in 8-bit quantization
- **Role**: Produces 2048-dimensional embeddings via masked mean pooling over the last hidden state
- **Parameters**: ~3B (entirely frozen — zero gradient updates)
- **Sequence length**: 512 tokens (see Section 5 for impact analysis)

Only the classification head (~7M parameters) is trained. This keeps training fast and prevents catastrophic forgetting of the backbone's code representations.

### 3.2 Handcrafted Features (102-dim)

Three complementary feature extractors run on the raw code string:

| Extractor | Dims | Captures |
|---|---|---|
| Stylometric | 12 | Indentation, spacing, naming conventions, comment density |
| Pattern Detector | 57 | AI-characteristic patterns, boilerplate, structural regularity |
| AST Extractor | 33 | Abstract Syntax Tree statistics, nesting depth, node type ratios |

These features capture signals that embedding-based representations may miss — e.g., consistent 4-space indentation, excessive docstrings, or unnaturally uniform variable naming, all characteristic of AI-generated code.

### 3.3 Fusion Head

The backbone embedding and handcrafted features are concatenated and passed through a small MLP with LayerNorm, GELU activations, and dropout (0.1), producing a 2-class logit output trained with standard cross-entropy loss.

## 5. Experimental Progression

### Attempt 1 — Baseline: Raw Competition Data, 128 Tokens
**Score: 0.31169**

The initial run used the raw SemEval competition training set (500,000 samples) with a sequence length of 128 tokens.

**Problems identified:**
- The raw dataset contained only **3 languages** (Python 91%, C++ 5%, Java 4%), creating a severe language bias
- Language balancing reduced the effective training set to ~57k samples by forcing equal counts per language, discarding 443k Python samples
- 128 tokens covered only **48% of samples fully** — more than half of all code snippets were truncated, losing content that often distinguishes human from AI code
- Training and validation reflected a near-balanced (52% AI) distribution, but the true test distribution differed, leading to miscalibrated predictions

---

### Attempt 2 — Increased Sequence Length: Raw Competition Data, 512 Tokens
**Score: 0.34630**

Increasing the sequence length from 128 to 512 tokens improved sample coverage from 48% to **87.5%**, providing substantially richer backbone representations.

**Improvement analysis:**
- Better backbone embeddings captured more complete code context
- The gain was modest (+3.5%) because the underlying dataset problems remained
- The model still trained on a 3-language dataset with systematic distribution biases

---

### Attempt 3 — Curated Dataset, 512 Tokens
**Score: 0.89382**

Switching to the curated SemEval dataset (described in Section 2) produced the decisive improvement.

**Why this dataset worked:**

| Factor | Raw Competition Data | Curated Dataset |
|---|---|---|
| Training samples | ~57k (after balancing) | 50,332 (all used) |
| Languages | 3 | **8** |
| Language balance | Forced equal (discards data) | Naturally balanced |
| Class balance | ~52% AI | **50% exactly** |
| Generator diversity | Narrow | **30+ models** |
| Balancing needed | Yes | No |
| Class weights needed | Yes (heuristic) | No |

The curated dataset directly addressed every failure mode of the original:

1. **8 balanced languages** → the model generalizes across programming paradigms, not just Python syntax
2. **Perfect 50/50 class balance** → no reweighting heuristics needed; the training signal is clean
3. **30+ generator sources** → the model learns generator-agnostic AI patterns rather than fingerprinting specific models
4. **All data used** → no samples discarded by artificial balancing

---

## 6. Impact of Data Quality vs. Model Complexity

The final score of **89.38%** was achieved without:
- Any class weighting
- Any language resampling
- Unfreezing the backbone
- Ensembling
- Threshold calibration

The ~58 percentage point jump from Attempt 1 to Attempt 3 came primarily from **data quality**, not model complexity. The architecture remained identical throughout all three attempts. This underscores a key principle: a well-designed model trained on poor data will underperform a simpler model trained on representative data.

---

## 7. Key Design Decisions

**Frozen backbone**: Keeps training lightweight (only 7M params updated) and prevents the model from unlearning StarCoder2's code representations. Embedding extraction is done once and cached.

**Handcrafted features alongside embeddings**: LLM embeddings capture semantic similarity but may normalize away surface-level stylistic signals. The 102-dim handcrafted features explicitly encode those signals, complementing rather than replacing the backbone.

**512 token limit**: A deliberate trade-off between coverage (87.5% of samples fully captured) and computational cost. 1024 tokens would cover 98.8% but roughly doubles embedding extraction time.

**No threshold calibration**: With a clean, balanced training set, the model's natural decision boundary (argmax at 0.5) is well-calibrated. Manually shifting the threshold based on assumed test distributions is a fragile, unprincipled approach that was explicitly avoided in the final submission.
